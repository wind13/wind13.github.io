Test 网络 说明

  • 个人或组织都可以注册 TestNet 的开发者账号。
  • 可以创建多个应用。
  • 每个应用有一个应用 ID（app_id），和一个或多个 key（app_key）。
  • 应用 ID 和 key 应该保密保管。
  • 如果一个开发者想让其他用户直接使用 ID 和 key 访问 BigChainDB Test Network，则必须获得明确的事先同意。

Test 网络 限制

  • 所有的应用受限于该应用所选的付费方案(plan)。
  • 目前，只提供了一种付费方案：免费方案，但有一些限制：每月 100 MB 流量，未来可能会修改及增加其他付费方案。
  • 所有应用每日访问次数总和不超过 500,000 API 次。
  • 所有应用每分钟访问次数总和不超过 600 API 次。
  • 单个 HTTP 请求内容(request body)大小(e.g. transaction size)不超过 15kB (15 kilobytes = 15000 bytes)， (我们使用 NGINX 设置 client_max_body_size 15k; 来实施这个限制)。
  • NGINX 被设置为每秒最大处理来自同一 IP 地址的 10 个请求。它可以在缓存中保留超过 10 个的请求以应对突然爆发的请求。

注：假设每个请求的大小是 1kb 的话，100 MB 的数据流量相当于大概 10 万个请求。

当一个应用超过 80%、90% 和 100% 的限额时，会给开发者发一封邮件，当一个应用超过它的限额（即 100%）时，则后面的请求将不允响应。

所有应用的数据限额在每个月的开始会被重置，如果一个应用在当月的限额没有用完，也不会延续到下一个月。

基础概念初步理解

Asset

类似于 Entity 实体，对应现实世界的某个与业务相关的人或物，也可以是可分拆的实体(Dividable Asset)，典型的案例就是虚拟货币 token 了。

实体一经创建(commit to block chain)就不可改变和删除了。

Metadata

源数据，保存该实体的相关动态数据，可以被不断更新，通常是与业务相关的一些状态数据等。

Transaction

一笔交易，或一次提交，可以分为创建(Create Transaction)和转移(Trasfer Transaction) 两种。更新 Metadata 可以通过转移交易来实现，即还是转移给原拥有者，但更新一下数据，源数据 Metadata 可以存各种类型的数据，类似 JSON 结构。

当前结论

区块链的应用还需要思考清晰相应的业务场景，同时也不清楚这样的架构是否能灵活支持业务上原来关系数据库常见的 CQRS/CURD 等操作，只能作为进一步研究的技术进行关注，目前还不能应用或测试。

也许，创建区块链应用时，不能再用传统的关系型数据库来思考业务建模了，需要调整为区块链的 Asset 和 Transaction 方式来思考业务建模了。
