REPL(Read Eval Print Loop:交互式解释器)

Chrome 开发者工具中的 Console 就是一个非常好的 REPL

服务端JavaScript

要实现在后台运行 JavaScript 代码，代码需要先被解释然后正确的执行。 Node.js 的原理正是如此，它使用了 Google 的 V8 虚拟机（ Google 的 Chrome 浏览器使用的 JavaScript 执行环境），来解释和执行 JavaScript 代码。

除此之外，伴随着 Node.js 的还有许多有用的模块，它们可以简化很多重复的劳作，比如向终端输出字符串。

因此， Node.js 事实上既是一个运行时环境，同时又是一个库。

进程与线程

  • 创建一个进程就相当于启动一个应用；
  • 创建一个线程则是在应用中分出去的运行分支；

同步与异步

  • 同步就是一件事交给别人做，但一直等他做完，拿到结果再去做下一件事；
  • 异步则是一件事交给别人做，自己去做别的事，等他完成拿到结果再回头做要做的事；

阻塞与非阻塞

  • 单线程会因同步而阻塞，需要用异步来实现非阻塞。

Callback hell

函数式编程中，函数是一等公民，可以被当作参数传入，或被当做结果返回。

不在事件循环中的回调

  • nextTickQueue & microTaskQueue: 即 process.nextTick() 和 promise() 任何时间完成后立刻执行；

事件循环中的顺序

  1. Timer: 执行所有能执行的 setTimeout, setInterval 定时任务；
  2. Pending i/o callback: 执行所有完成了的 callback 和 error handle 方法；
  3. Idle, Prepare: node 做一些内部操作；
  4. Poll: 文件读响应，新的 socket 或者 http 连接请求等；
  5. Check: 执行所有能执行的 setImmediate 任务；
  6. Close: 执行所有能执行的 close callback 任务，如果无任务了则退出；

Express.js 简介

  • Express 不对 Node.js 已有的特性进行二次抽象，只是在它之上扩展了 Web 应用所需的基本功能。
  • Express 是一个自身功能极简，完全是由路由和中间件构成一个的 web 开发框架：从本质上来说，一个 Express 应用就是在调用各种中间件。
  • 不会了就查 Express 4.x API 中文手册

中间件（Middleware）

  • 一个有三个参数：req, res, next 的函数。
  • 中间件的功能包括：
    • 执行任何代码。
    • 修改请求和响应对象。
    • 终结请求-响应循环。
    • 调用堆栈中的下一个中间件。
  • 中间件最终结果（二选一）：
    • 终结请求-响应循环
    • 调用 next() 方法
  • 第三方中间件

中间件分类

  • 应用级中间件: app.xxx
  • 路由级中间件: router.xxx
  • 错误处理中间件: 必须 (err, req, res, next) 四个参数
  • 内置中间件: 唯一内置 express.static(root, [options])
  • 第三方中间件: app.use(xxx());

fs 文件系统 一

  • 大部分 fs 操作接受字符串、Buffer、或使用 file: 协议的 URL 对象作为文件路径。
  • 所有文件系统 API 中，除了 fs.FSWatcher() 和那些显式同步的方法之外，都使用了 libuv 的线程池。
  • 成功调用 fs.watch() 方法会返回一个新的 fs.FSWatcher 对象。
  • 成功调用 fs.createReadStream() 会返回一个新的 fs.ReadStream 对象。
  • fs.Stats 对象提供了一个文件的信息。stats.isDirectory() 是否文件系统目录。

Note:

  • 在 Windows 上 Node.js 遵循 per-drive 工作目录的理念。例如，fs.readdirSync(‘c:\‘) 可能会返回与 fs.readdirSync(‘c:’) 不同的结果。
  • 在 POSIX 系统，内核为所有进程维护着一张当前打开着的文件与资源的表格。因此打开用完后要记得关闭，否则长期大量累积会造成内存溢出系统崩溃。
  • 在 ECMAScript 2015 (ES6) 引入 TypedArray 之前，JavaScript 语言没有读取或操作二进制数据流的机制。 Buffer 类被引入作为 Node.js API 的一部分，使其可以在 TCP 流或文件系统操作等场景中处理二进制数据流。
  • Buffer 类在 Node.js 中是一个全局变量，因此无需使用 require(‘buffer’).Buffer。

–

fs 文件系统 二

  • fs.createReadStream(path[, options]) 新建一个 fs.ReadStream 可读流。
  • fs.createWriteStream(path[, options]) 新建一个 fs.WriteStream 可写流。
  • fs.exists 已被 fs.stat() 或 fs.access() 代替。
  • 最简单的方式就是写个小示例去试试，同时需要补一下 Linux 文件相关知识和命令。

Note:

参考文档

  • qize的博客

  • NodeJs之fs的读写删移监

  • 不建议在调用 fs.open() 、 fs.readFile() 或 fs.writeFile() 之前使用 fs.access() 检查一个文件的可访问性。 如此处理会造成紊乱情况，因为其他进程可能在两个调用之间改变该文件的状态。 作为替代，用户代码应该直接打开/读取/写入文件，当文件无法访问时再处理错误。

Node.js 异步错误处理问题

  • 将错误从底层不断地向上层传递，直到错误到达某个层级有足够的信息去决定如何处理这个错误。
  • 用 try ... catch 实现异常处理流程。
  • 总是抛出一个继承自 Error 的对象，并附加上下文的属性。
  • 无法在异步代码中直接使用 try ... catch 来捕捉异常。Callback 中需要手动完成错误传递等。

–

Node.js 异步错误解决方案

  • Promise 可以说是对同步任务和异步任务的一种一致的抽象。尽量避免手动创建 Promise，使用 Promise.promisify 函数自动完成转换。
  • Promise、co/generator 和 asycn/await 谁才是未来？
  • EventEmitter 则提供了一个操作（甚至没有操作）对应多个结果的异步模型。
  • 终极错误捕捉手段：process.on('uncaughtException') 和 process.on('unhandledRejection')

Note:

  • 参考 Node.js 错误处理实践

错误处理的最佳实践

  • 注意 Promise / callback chain 不要从中间断开
  • 只处理已知的、必须在这里处理的异常，其他异常继续向外抛出
  • 不要轻易地丢弃一个异常
  • 传递的过程中可以向 err 对象上添加属性，补充上下文
  • 如果还要继续抛出这个错误，就不要轻易打印日志
  • 错误汇总到 边界 处，用统一的、健壮的方式去处理
